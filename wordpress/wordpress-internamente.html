FUNCIONAMENTO INTERNO E PONTO DE ENTRADA

Toda requisição enviada para uma instalação do Wordpress irá percorrer, inicialmente, o mesmo caminho. Outras requisições, muito provavelmente, serão bloqueadas pelo servidor a fim de proteger a estrutura de diretório da nossa instalação.

1 - PONTO DE ENTRADA GERAL DO WORDPRESS

Na maior parte dos servidores que estão rodando instâncias do Wordpress, o primeiro arquivo de programação a ser interpretado, será o .htaccess.

O arquivos .htaccess contém instruções e diretrizes para o Servidor Web Apache, ou seja, dentro do .htaccess é onde estão contidas as principais regras de como o Servidor deve se comportar mediante diversas requisições. Então, ao encontrar um arquivo .htaccess o Servidor Apache irá enviar para o mesmo a URL oriúnda de um cliente da Web. Após o script receber a URL o mesmo pode fazer com que o Apache aja de uma determinada maneira, para as mais diversas situações que podem vir a ocorrer.

Por exemplo, podemos definir no arquivo .htaccess que um diretório foi alterado, ou então, que um diretório não pode ser servido (enviado para o cliente). Pode verificar se a URL em questão está requisitando um arquivo de mídia e direcionar a requisição para o diretório onde estão nossas imagens, vídeo, arquvos CSS e etc. Inclusive, podemos permitir que um diretório esteja disponível para um determiando IP, e para uma determianda faixa de IP, o diretório em questão naõ deve estar disponível.

# BEGIN WordPress
<IfModule mod_rewrite.c>
RewriteEngine On
RewriteBase /wordpress/
RewriteRule ^index\.php$ - [L]
RewriteCond %{REQUEST_FILENAME} !-f
RewriteCond %{REQUEST_FILENAME} !-d
RewriteRule . /wordpress/index.php [L]
</IfModule>

# END WordPress

Acima, temos o código que vem por padrão na versão 4.2 do Wordpress. Como é possível ver, está sendo definindo pontos de entrada, ou seja, arquivos de extensão .php que devem ser, por padrão, executados.

PONTO DE ENTRADA DOS TEMAS

Um tema possui uma lógica que define qual será o ponto de entrada, ou seja, qual o arquivo que o Wordpress irá invocar para que a programação da montagem do tema seja executada.

A LÓGICA DE ENTRADA

Quando o Wordpress recebe uma solicitação de exibição de página, o arquivo index.php, contido na raiz de cada instalação do Wordpress é inicializado. Em seguida, o mesmo invoca funções que irão verificar qual o tema definido pelo usuário na parte adminstrativa do Wordpress e então, irá verificar se a pasta do tema ativo contém um de 3 arquivos, são eles:
  1. front-page.php
  2. home.php
  3. index.php

2 - ARQUIVO index.php

O objetivo do .htaccess é direcionar todas as requisições GET para o index.php. Se observarmos atentamente os outros arquivos *.php contidos na raiz da pasta do Wordpress, veremos que estes não compõem outras URL, ou seja, você provavelmente nunca viu um link com o nome de algum desses arquivos - salvo quando estavas instalando Wordpress.

Abaixo, temos o código contido no arquivo index.php. As descrições foram traduzidas de maneira não literal.

<?php
/**
 * Frente da aplicação Wordpress. Este arquivo não faz nada, mas abre
 * o arquivo ``wp-blog-header.php`` e este executado e diz para o     * Wordpress abrir o tema.
 *
 * @package WordPress
 */

/**
 * Diz para o WordPress abrir um tema e envia-lo.
 * @var bool
 f*/
define('WP_USE_THEMES', true);

/** Inicia o Ambiente Wordpress e seus Templates*/
require( dirname( __FILE__ ) . '/wp-blog-header.php' );

Como visto, o arquivo de entrada, ou seja, o````index.php`` serve somente como um ponto de entrada e nada mais. O mesmo não faz qualquer roteamento e não toma nenhuma decisão.

A primeira instrução do arquivo index.php é definir o tema ativo e a função require() abre e executa o arquivo wp-blog-header.php. A partir desse momento, podemos dizer que o WordPress foi realmente iniciado. Então, vamos estudar o código que será executado:

<?php
/**
 * Abre o Ambiente Wordpress e seus Templates
 * @package WordPress
 * /

if ( !isset($wp_did_header) ) {

        $wp_did_header = true;

        require_once( dirnameFILE__) . '/wp-load.php' );

        wp();

        require_once( ABSPATH . WPINC . '/template-loader.php' );
}

Inicicialmente, é verificado se a variável $wp_did_header está setada. Caso esteja, isso significa que que alguns cabeçalhos já foram enviados e assim, evita que o WordPress seja iniciado mais de uma vez.

No caso afirmativo, ou seja, caso a variável $wp_did_header não esteja definida no ambiente, então a mesma é definida e inicializada com o valor True.

Em seguida, o arquivo wp-load.php é inicializado e quando executado o mesmo irá procurar pelo arquivo wp-config.php. Neste é onde foram definidas as constantes para a conexão com o banco, para a autenticação de usuários, depuração e etc.

Num primeiro momento, é verificado se o arquivo wp-config.php existe na pasta de instalação do Wordpress, ou então, é verificado se o arquivo wp-config.php existe no diretório acima da pasta onde está instalado porém, e também, é verificado se nesse nível acima NÃO existe o arquivo de nome wp-settings.php. Abaixo temos os 2 diretórios buscados que extraí da depuração.

htdocs/wp-config.php
htdocs/wordpress/wp-config.php

Na segunda condição, ou seja, quando o arquivo wp-config.php não está na pasta de instalação do Wordpress, mas o mesmo existe a um nível acima, é necessário verificar, se nesse nível acima não existe uma outra instalação do Wordpress. Senão, passaríamos a executar o arquivo de configuração de uma outra instação.

Trocando em miúdos, uma das 2 condições a seguir precisam ser verificadas para que o arquivo wp-config.php seja aberto e executado:
  1. O arquivo existe na pasta de instalação
  2. O arquivo existe num diretório acima e neste NÃO existe o arquivo wp-config.php, que sugeriria uma outra instalação do framework.

Por fim, caso nenhuma condição seja satisfeita, é executado a instalação do Wordpress, até porque, o mesmo entende que precisa ser instalado para que esses arquivos sejam criados.

Nas versões antigas do Wordpress, éramos obrigados a copiar esse arquivo no braço, até porque, havia alguma restrição que não permitia o Wordpress a escrever no diretório root da aplicação.

Em seguida, a função wp() que está definida no arquivio wp-include\function.php é executada e nenhum parâmetro é enviado. Essa função, tem o objetivo de registrar como variável de ambiente qualquer valor que seja enviado como parâmetro.

Neste caso, executamos a função wp() somente para que as variáveis de ambiente sejam inicializadas, nada mais. Então, a função wp() não faz nada especial e não retorna valores, apenas define o valor enviado como parâmetro como sendo uma variável de ambiente.

Por fim, o arquivo template-loader.php é aberto e executado e então, o WordPress abrirá o tema definido como padrão e irá invocar, uma a uma, as funções para a renderização do nosso tema.
