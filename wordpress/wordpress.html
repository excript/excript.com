NOME DURAÇÃO

ASSETS

No desenvolvimento WEB, Assets representam todos os arquivos que compõem o nosso tema. Por exemplo, comumente um site possui arquivos de CSS, JavaScripts, imagens, fontes especiais e outros arquivo referentes a tecnologia empregada. A esses arquivos, damos o nome da Assets.

ASSETS NO WORDPRESS

O Wordpress centraliza e administra os assests de cada tema, e assim, os mesmos são adicionados ao código através das funções wp_enqueue_style e wp_enqueue_script que estão definidas no arquivo functions.php.

Outros assets serão baixados a cada vez que o site for visualizados. Estes arquivos, são armazenados num serviço de nome CDN.

CDN .

O acrônimo CDN é oriundo de Content Delivery Network que, numa tradução livre, significa Rede de Distribuição de Conteúdo.

Uma Rede de Distribuição de Conteúdo são servidores comerciais que armazenam arquivos estáticos de bibliotecas e/ou frameworks que são utilizados por um grande número de aplicações.

Cada CDN possui computadores nos principais núcleos da internet a fim de estar o mais próximo possível do usuário. Ao invés de manter os dados centralizados, sobrecarregando um único servidor e trafegando sempre as mesmas informações, esse modelo de serviço responde ao usuário do nó (computador) mais próximo.

Por fim, uma CDN deve ser entendida como servidores que mantem cópias de informações em diversas regiões do planeta para entregar rapidamente as requisições e também, evitar que uma mesma informação seja trafegada pela rede inúmeras vezes de um extremo ao outro da grande rede.

A utilização de CDN diminui a latência, melhora o paralelismo e o armazenamento em cache.

Algumas CDN famosas são:
  • bootstrap CDN - http://www.bootstrapcdn.com/
  • Google Hosted Libraries (Google API) - https://developers.google.com/speed/libraries/

UTILIZANDO A API DO WORDPRESS

O desenvolvimento para Wordpress ocorre de maneira procedural, ou seja, o Wordpress disponibiliza uma série de funções e o nosso trabalho é invocar as funções que precisamos para extrair informações.

O conceito elementar na construção de um tema, é entender, no mínimo, o básico da hierarquia de templates.

CASO DE ESTUDO

Vamos supor uma situação qualquer, onde o Wordpress recebeu uma solicitação. Este, irá definir a precedência de template baseando-se na URL recebida. Então, vamos supor agora que o Wordpress recebeu uma requisição para que seja exibido a página inicial do site.

Inicialmente, a tomada de decisão para que seja renderizado a página inicial, se dá quando nenhuma query é enviado para o Wordpress.

Nessa situação, o Wordpress irá checar qual o tema que está ativo e então, tentará invocar o template front-page.php contido dentro da raiz do tema que está ativo. Caso o arquivo não exista, o Wordpress irá procurar pelo template home.php. Se o mesmo existir, a página é montada e o resultado enviado como respostado. Do contrário, ou seja, caso esse template também não exista, será invocado o template index.php.

Como estudado, o template index.php é necessário para que um determinado tema funcione, do contrário, será levantado uma exceção e uma mensagem de erro será retornada para o cliente.

Este é a grosso modo como o Wordpress processa as solicitações. É importante observar que inicial, o Wordpress não exige que exista um único código no template index.php, a exigência é que este arquivo exista na raiz de cada tema.

INVOCANDO FUNÇÕES DA API

Abra o seu arquivo index.php e cole o código a seguir:

<?php bloginfo('name'); ?>

Em seguida, certifique-se que o template ativo nas configurações do Wordpress seja o mesmo do local onde você editou o arquivo index.php. Em seguida, digite a URL da instalação do seu Wordpress e será impresso na tela, o nome do site que foi definido no Admin do Wordpress.

A função bloginfo() está definida no arquivo general-template.php e a mesma invoca a função get_bloginfo() onde está definido o que deve ser retornado para cada parâmetro que seja enviado. Assim, o objetivo da função é retornar informações do nosso site. A função bloginfo() tem definido um parâmetro tipo string, no caso, há definido várias constantes e cada uma faz com que a função retorne um tipo de informação diferente. A seguir, temos a lista de parâmetros possíveis e o respectivo valor que será retornado:

  • quando a função bloginfo() é invocada e nenhum parâmetro é enviado, assume-se que o parâmetro é 'name', ou seja, é retornado o nome do site que foi definido na área administrativa.
  • url: http://localhost/wordpress

    Retorna a url da home do site. A mesma informação pode ser obtida com a função home_url().

  • wpurl: http://localhost/wordpress

    retorna a url especificamente do nosso site. A mesma informação pode ser obtida através da função site_url().

  • description:

    Só mais um site WordPress

  • rdf_url: http://localhost/wordpress/feed/rdf/

  • rss_url: http://localhost/wordpress/feed/rss/

  • rss2_url': http://localhost/wordpress/feed/

  • atom_url: http://localhost/wordpress/feed/atom/

  • comments_atom_url: http://localhost/wordpress/comments/feed/atom/

  • comments_rss2_url: http://localhost/wordpress/comments/feed/

  • pingback_url: http://localhost/wordpress/xmlrpc.php

  • stylesheet_url: http://localhost/wordpress/wp-content/themes/amk-primer-tema-wordpress/style.css

  • 'template_directory' template_directory: http://localhost/wordpress/wp-content/themes/amk-primer-tema-wordpress

  • version: 4.2.4

Tudo, sem exceção, que é definido na área Admin do Wordpress pode ser acessado de alguma maneira no desenvolvimento do tema, e assim, a única coisa que precisamos, é conhecer as principais funções para montarmos, dinamicamente, o tema quando o Wordpress solicitar.

VARIÁVEIS DE AMBIENTE DO WORDPRESS

Se colocarmos um breakpoint em qualquer ponto de entrada do nosso tema, conseguiremos obter a lista de variáveis definidas pelo Wordpress. Se por exemplo, verificarmos as variáveis de ambiente no template index.php, será possível saber qual usuário está logado. Para obter essa informação, basta inspecionar a variável $current_user. Outra variável interessante, dizem respeito ao navegador do cliente que enviou a solicitação, para isso, temos várias variáveis, por exemplo:

  • $is_IE
  • $is_chrome
  • $is_gecko
  • $is_iphone
  • $is_lynx
  • $is_macIE
  • $is_opera
  • $is_safari
  • $is_winIE

Também podemos inspecionar as query SQL que foi enviada para o banco, através da variável $wp_query. E conseguirmos saber não somente o SQL que foi envia e a respectiva resposta, há muito mais informações que podemos explorar para entendermos melhor o Wordpress ou mesmo, para fazermos alguma interrupção ou injetarmos algum código.

Provavelmente, a variável mais importante e que podemos extrair muitas informações é a $post. Nos templates index.php e home.php, o wordpress disponibiliza funções para iterarmos as postagens, assim, temos que a variável $post sempre irá conter uma postagem e após invocarmos a função the_post(), avançaremos para o próxima postagem e assim, se inspecionarmos a variável $post novamente, veremos que há outra postagem selecionada.

Enquanto a variável $post, no singular, contém um único postagem, também há disponível a variável $posts, onde encontramos um Array com todas as postagens que foram retornadas.

ARQUIVO functions.php

Um tema, pode fazer uso de um arquivo de nome functions.php que é aberto automaticamente pelo WordPress. Este arquivo, deve estar contido na raiz do nosso tema e o seu nome deve ser exatamente functions.php.

Em suma, devemos pensar no arquivo functions.php como se fosse um plug-in, onde podemos adicionar funções globais e enfileirar outras funções para que estas sejam abertas conforme a mesma foi adicionada na fila para a sua abertura.

É normal que o arquivo functions.php enfileirar a abertura dos arquivos CSS, JavaScript, Fontes e etc. Para esses arquivos estáticos, podemos fazer uso das funções wp_enqueue_scripts() e wp_enqueue_style().

Também é comum registrarmos as Sidebars, os Menus de Navegação, Formatos de postagens, Cabeçalhos de páginas personalizados, Backgrounds personalizados e muitas outras funções que viermos a necessitar no desenvolvimento.

Também é comum a adição de funções globais, assim, toda função adicionada no arquivo functions.php poderá facilmente ser acessada de qualquer arquivo PHP pertencente ao nosso tema.

SIDEBAR

Sidebar são as zonas do tema em que é possível adicionar widgets através da parte administrativa do Wordpress. O Wordpress define vários widgets por padrão, porém, é possível, através de plug-ins, personalizar widgets ou então, criar um novo a partir do zero.

É interessante que você acessa a área de Widgets na Dashboard antes de implementarmos os códigos da Sidebar e observe como a mesma está desabilitada. Isto é, o Wordpress mantém os Widgets desabilitados por padrão, logo, esta área só será ativada a partir do momento em que registrarmos uma zona de Sidebar em nosso tema.

SIDEBAR E WIDGETS

Sidebar e widgets são conceitos distintos, porém, um é complemento do outro. No desenvolvimento de um tema temos que escolher as regiões em que será possível a adição de widgets.

Por padrão, a sidebar deve ser definida no arquivo sidebar.php. Assim, temos que pensar neste arquivo PHP da mesma forma como estudamos os arquivos header.php e footer.php.

Se partimos da premissa que a nossa Sidebar estará localizada no canto direito do vídeo, e que no centro, estamos utilizando um sistema de grid que ocupam 8 unidades, poderíamos definir a programação da Sidebar da seguinte maneira:

<div class="col-md-4">
    <?php get_sidebar(); ?>
</div>

Estamos definindo que a região da barra-lateral utilizará 4 unidades quando estivermos renderizando o nosso site numa resolução de vídeo de tamanho md ou superior.

Em seguida, invocamos a função get_sidebar() que desempenhará a mesma função que as funções get_header() e get_footer().

REGISTRANDO ZONAS DINÂMICAS EM functions.php

A seguir, temos o Array que podemos passar como parâmetro pela função register_sidebar().

array(
    'name'          => ... , //Nome da área dinâmica
    'id'            => ..., //ID da zona da sidebar
    'description'   => ..., //Descrição da sidebar
    'before_widget' => ..., //HTML que precederá o Widget
    'after_widget'  => ..., //HTML que sucede o Widget
    'before_title'  => ..., //Tag a ser inserida antes do título
    'after_title'   => ... //Tag a ser inserida depois do título
));

CÓDIGO PARA O REGISTRO DA SIDEBAR

A seguir, temos o código a ser inserido no arquivo functions.php.

php
//função builtin que registra zonas dinâmicas (sidebar)
register_sidebar(array(
    'name' => __('Lateral direita', 'toinnn'), //Nome da área dinâmica
    'id' => 'lateral-direita', //ID da zona da sidebar
    'description' => __( 'Zona dinâmica do tema (sidebar)', 'toinnn'), //Descrição da sidebar
    'before_widget' => '<div id="%1$s" class="widget %2$s">', //HTML que precederá o Widget
    'after_widget'  => '</div>', //HTML que sucede o Widget
    'before_title' => '<h3>', //Tag a ser inserida antes do título
    'after_title' => '</h3>' //Tag a ser inserida depois do título
));

ACESSANDO O DASHBOARD (área administrativa)

Se o registro da sidebar no arquivo functions.php foi realizado corretamente, a área para adicionarmos Widgets será ativada automaticamente. Então, para que você tenha a certeza de que a sua Sidebar está funcionando corretamente, acesse a área administrativa e veja se a área de widgets está ativa.

ARQUIVO sidebar.php

Normalmente o código contido no arquivo sidebar.php é bastante simples.

Inicialmente, temos que verificar se a área que definimos como sendo da Sidebar está ativa. Para isso, o Wordpress disponibiliza a função is_active_sidebar(). Como função, basta passarmos o ID que definimos como sendo da Sidebar.

Caso a área esteja ativa, então, basta invocarmos a função dynamic_sidebar(), passando novamente a ID que foi definida como sendo da Sidebar e que verificamos que a mesma está ativa como parâmetro. Feito isso, o núcleo do Wordpress tratará de renderizar, da melhor maneira, os Widgets que o usuário do nosso tema adicionou.

A seguir, temos o código completo que devemos colocar no arquivo sidebar.php:

<?php
    if( is_active_sidebar('lateral-direita') ) { // Verifica se a área "lateral-direita" esta ativada
        dynamic_sidebar( 'lateral-direita' ); // Em caso afirmativo, invocamos a função que abre os Widgets
    }
?>
